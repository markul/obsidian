## Findings

- The repo already had empty directories for alternative local CronJob layouts, which suggests this split was anticipated but unfinished
- `devops/Dockerfile` previously published only the API project; a direct publish check confirmed the runtime output lacked `Skp.ProductChangeWorkflowService.CronJob.dll`
- After the Docker and publish changes, the publish output contains both `Skp.ProductChangeWorkflowService.Api.dll` and `Skp.ProductChangeWorkflowService.CronJob.dll`
- The first `Dev` config attempt was over-scoped: it mixed local-skaffold and remote-dev assumptions and made the API and CronJob loaders inconsistent
- The corrected shape is:
  - local skaffold API uses the local config directly
  - developer remote-dev execution uses `Local + Dev overlay`
  - the `api` profile now starts both the API deployment and CronJobs
  - there is no standalone CronJob skaffold profile anymore
  - in local k8s only the API pod uses Kafka, and it continues using `localhost:9092`
- The final verification gap was not in skaffold profile wiring; it was in the Docker image build path for the newly added CronJob publish step
- The extracted integration scenario still expects in-pod access to the Kubernetes API, so restoring the lightweight `kubectl-proxy` sidecar plus RBAC was the smallest compatible change

## Decisions

- Use separate `Local` and `Dev` YAML variants instead of forcing one config file to serve both host-local and in-cluster skaffold execution
- Keep `Dev` as a developer-only override file, not as a required skaffold runtime config
- Keep CronJob support behind explicit skaffold profiles so the default dependencies-only flow stays lightweight
- Keep the local skaffold surface at only two modes: default dependencies-only and `api` for dependencies plus API plus CronJobs
