## Key Decisions

- keep `appsettings.Dev.yaml` as a developer-only overlay instead of making skaffold depend on it at runtime
- keep the local skaffold contract to two modes only: default dependencies and `api` for dependencies plus API plus CronJobs
- keep the Kubernetes CronJob resource names lowercase DNS-style while the process-pending job executes the existing `ProductChangeRequestProcessingWorker` command inside the container
- keep the default dependencies-only flow unchanged, and make `api` the only profile that starts API plus CronJobs
- keep the imported integration scenario compatible with the current strategy by adding only the in-pod Kubernetes access support it needs, not by reintroducing a separate CronJob image, overlay, or skaffold profile
- do not add CronJob support back into the Skaffold integration-test path: the shared Jenkins pipeline is effectively limited to a single deployment-style manifest for integration tests, so this ticket deliberately keeps the local/Jenkins pod shape as one `Deployment` and excludes CronJob resources from the supported Skaffold/Jenkins flow
- keep the Jenkins smoke test on an HTTP route that is definitely mapped in the API and can prove the stage reaches the service; `GET /process/v10/run` returning `405` is acceptable because it confirms the API is reachable on the intended route while avoiding the downstream permission-token failure path currently seen on `POST /process/v10/run`
- accept that the current `GET /process/v10/run` smoke test is a stage-availability check, not an endpoint-authorization check: local verification showed the same `405` is returned even without an `Authorization` header, so the bearer-token hook is exercised but the route itself does not prove auth enforcement
- keep the bearer token entirely out of `Molder:IntegrationTests` static config placeholders; Molder treats config-loaded variables as global and they are awkward to override, so the integration-test hook should create `Authorization` dynamically instead of trying to replace a preloaded global key
- keep the local authorization Thrift mock in a dedicated image so its transitive dependency graph cannot overwrite API runtime assemblies inside the main service image
- align the product-change integration-test image trust store with `limit-service` by vendoring the same TestOps CA certificates into this repo instead of assuming the base SDK image already trusts the internal CA chain
