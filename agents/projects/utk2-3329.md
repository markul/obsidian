## Current State

- Jira `UTK2-3329` is currently `Backlog` and asks for adequate logging of all inbound and outbound requests in `skp-product-change-workflow-service`
- API startup now enables `Skp.Logging` detailed request logging through `AddRequestLogging(..., x => x.DetailedLogEnabled = true)`
- API middleware order now runs `UseRequestLogging()` after `UseRouting()`, so endpoint metadata is available and `LogBodyAttribute` can actually affect inbound body logging
- `StepController.RunAsync` keeps `LogBody(IgnoreLogBodyOptions.IgnoreResponse)` because it can return the large serialized workflow context; the smaller process and event endpoints no longer opt into body logging
- Infrastructure client registration now uses a local `OutboundHttpLoggingDelegatingHandler` instead of `CustomLoggingHttpMessageHandler`, so outbound logs no longer enrich `httpHeaders`
- The new outbound handler preserves method, URI, status, elapsed time, and optional request or response body fields, logs under the existing `AlfaBank.Http.Message` category, and now truncates logged bodies with `HttpPayloadLogging.MaxLoggedBodyLength`
- Local validation is green: `dotnet build -m:1` passed and the full unit-test project passed with `253` tests green
- Unit coverage now includes focused tests for the outbound handler response logging, exception logging, and payload truncation behavior
- The stale autogenerated Reqnroll file `001.TransferExpiringDealToProductChangeRequestService.feature.cs` was removed, so the solution build is back to `0` warnings

## Key Decisions

- Reuse the existing `Skp.Logging` inbound middleware, but replace the packaged outbound HTTP message handler with a repo-local handler because the package unconditionally logs request headers
- Keep request-body logging only on the endpoint with the potentially large workflow-context response, and skip its response-body logging to avoid duplicating that payload in logs
- Keep the outbound logger category as `AlfaBank.Http.Message` so existing log searches do not need to change
- Use a repo-local `HttpPayloadLogging.MaxLoggedBodyLength` option for outbound body truncation; the inbound `Skp.Logging` package still exposes only `DetailedLogEnabled`
- Keep request logging before authentication but after routing so unauthenticated requests are still logged while endpoint metadata remains available for body-logging decisions
